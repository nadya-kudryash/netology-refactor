# netology-refactor

Требования к рефакторингу
Примените Single Responsibility Principle:
· Разделите ответственность за работу с БД и бизнес-логику
· Выделите отправку уведомлений в отдельный компонент

Реализуйте Open/Closed Principle:
· Сделайте систему расширяемой для добавления новых типов баз данных и уведомлений

Примените Liskov Substitution Principle:
· Создайте базовый интерфейс для работы с сообщениями
· Реализуйте разные типы отправителей сообщений

Реализуйте Interface Segregation Principle:
· Разделите большие интерфейсы на мелкие, если требуется
· Создайте отдельные интерфейсы для инициализации базы данных (создание таблиц) и записи

Примените Dependency Inversion Principle:
· Внедрите зависимости через интерфейсы
· Используйте внедренные зависимости

Инструкция к выполнению
Создайте абстракции (интерфейсы).
· Создайте интерфейсы RepositoryWriter и Notifier, которые будут определять общие контракты для всех типов баз данных и отправителей уведомлений.
Адаптируйте существующие структуры под интерфейсы.
· Выделите структуру EmailSender так, чтобы она реализовывала интерфейс Notifier. Для этого нужно перенести метод sendEmailNotification в нее и переименовать его в Send.
Добавьте новую функциональность.
· Создайте новую структуру SMSSender, которая также будет реализовывать интерфейс Notifier. Это продемонстрирует гибкость новой архитектуры.
4. Проведите рефакторинг основного сервиса.
· Создайте OrderService так, чтобы он зависел от интерфейсов RepositoryWriter и Notifier, а не от конкретных реализаций.
5. Продемонстрируйте работу.
· В функции main покажите, как можно использовать OrderService с разными типами отправителей (EmailSender и SMSSender), не меняя код самого сервиса.
6. Загрузите решение на GitHub
· Создайте новый публичный репозиторий на GitHub.
· Загрузите в него получившиеся файлы с итоговым решением.
Вставьте ссылку на вашу работу в поле «Ссылка на решение» и нажмите «Отправить на проверку». Перед отправкой домашнего задания вы можете написать комментарий эксперту.

Критерии оценки домашнего задания
«Зачёт»
· SOLID-принципы применены: код успешно отрефакторен. OrderService зависит от абстракций (RepositoryWriter, Notifier), а не от конкретных типов
· Интерфейс используется корректно: созданы интерфейсы RepositoryWriter и структура SQLiteRepo, его реализующая, а также Notifier и реализующие его структуры EmailSender, SMSSender
· Гибкость системы: в main продемонстрирована работа сервиса с разными отправителями (email и sms)
· Код работает: программа компилируется и выполняется без ошибок, вывод в консоль соответствует ожиданиям
· Читаемость: код хорошо отформатирован (gofmt) и понятен
· Решение на GitHub: ссылка на репозиторий с кодом предоставлена
По желанию:
· Дополнительным плюсом будет правильная организация пакетов, с разделением на слои и использованием каталога internal
· Если получится, попробуйте написать один-два теста, в которых RepositoryWriter или Notifier реализован в виде «мока» — например, MockWriter, который просто отработает без записи в базу, чтобы тест не зависел от ошибок базы данных
